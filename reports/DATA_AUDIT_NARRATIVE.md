## 2. Китай — активная панель (основной тренировочный датасет)

**Файл:** `data/raw/china/active/Wind Software & services.xlsx` (644 KB, 1 лист «Sheet1»)

### 2.1 Что это
Выгрузка из WIND по отраслевому срезу «Software & Services»: 319 действующих компаний, период 2014–2025, 15 финансовых метрик. По переписке и по факту — **центральный тренировочный набор** для китайской руки исследования (против которой будет сравниваться российская выборка из SPARK).

### 2.2 Как это устроено (нестандартно — внимание)
Формат файла **не соответствует** типичному `(company × year × metric)` панельному представлению. Сырая форма:

```
Размерность: 199 строк × 321 столбец
Столбцы:      代码 | Code | <319 тикеров компаний, например 000004.SZ, 000032.SZ, …>
Строки:       метки вида «Total Revenue\n[unit]1M\n[rptDate]2014-12-31\n[rptType]Consolidated»
              (15 метрик × 12 лет = ~180 строк + заголовочные строки Name/代码/Code,
              повторяющиеся между блоками)
Ячейки:       значение метрики для конкретной (компания, метрика, год)
```

То есть **тикеры лежат в столбцах, а не в строках**. Это выгрузка из интерфейса WIND «как есть», без дальнейшего unpivot. Для работы её нужно однократно канонизировать в длинную форму `(ticker, year, metric, value)` — я уже написал парсер в [`notebooks/data_audit.py`](../notebooks/data_audit.py), функция `parse_wind_file`, он даёт 57 420 наблюдений длинной панели.

### 2.3 Что работает хорошо

Общая заполненность длинной формы — **71.5 %** non-null (41 043 из 57 420). По **метрикам** картина такая (в порядке убывания покрытия, % non-null через все тикеры-годы):

| Метрика | Покрытие |
|---|---:|
| Total Assets | 94.8 % |
| Total Revenue | 90.2 % |
| Total Liabilities | 87.0 % |
| Operating Cash Flow (`net_cash_flows_oper_act`) | 86.8 % |
| Beginning Cash & Equivalents | 86.8 % |
| Cash Received from Sales | 86.8 % |
| Total Shareholders' Equity | 86.8 % |
| Total Current Assets | 86.8 % |
| Intangible Assets | 83.8 % |
| EBITDA | 83.1 % |
| EBIT | 83.1 % |

Эти 11 метрик — **фундамент feature engineering**. Из них выводятся все ключевые предикторы гипотез H1–H3: ликвидность (Cash, Current Assets, OCF), innovation capacity (Intangibles, R&D — см. ниже), размер (Assets, Revenue), капитал-структура (Assets vs. Liabilities vs. Equity).

По **годам** заполненность растёт с 60.6 % (2014) до 85.7 % (2024) — типично для выгрузок из WIND: старые отчётности хуже оцифрованы, свежие заполнены системно. Это означает, что модели на окне 2018–2024 будут обучаться на почти полной панели (>80 %), а более ранние годы пригодятся скорее для TTC-калибровки / historical macro feature lags.

### 2.4 Что сломано или требует внимания

**(а) R&D и Short-term Loans — частично пусто в ранних годах.**
R&D начинает появляться только с 2018 (до этого 0–17 %). Это важно для H2 («intangibles важнее долга»): фича существует, но не для всего исторического окна. Решение — либо ограничить временное окно с 2018, либо forward-fill внутри компании.

Short-term Loans — 53.5 % в среднем, 44–69 % по годам. Использовать можно, но нужно помнить, что это прокси долга (не полный Total Debt).

**(б) Interest Expense — практически отсутствует (1.2 %).**
В чате команды это уже было отмечено («практически отсутствует interest expense») и теперь подтверждено цифрой. Покрытие по годам: 0.3–3.4 %. **Фича непригодна** — строить Interest Coverage Ratio по этой выборке нельзя. Придётся обходиться без метрики стоимости обслуживания долга для китайской руки (и компенсировать Short-term Loans + размером долга через Total Liabilities).

**(в) Net Profit of Current Period — 0 % заполнения. Блокер.**
В WIND-дампе для этого поля указан код `anal_reits_netprofit`, и он **полностью пуст** во всём файле. Это, по всей видимости, неверно выбранное поле при выгрузке — вероятно, оно специфично для REIT-секции (см. префикс `anal_reits_`), а не для software & services. **Чистая прибыль — критически важный показатель** (для ROA, ROE, маржи). Без неё невозможно построить ни один из классических financial distress признаков (Altman Z-score baseline, Net Income / Assets и т.д.).

**Что нужно сделать:** при следующей выгрузке из WIND заменить `anal_reits_netprofit` на корректный код `net_profit_is` (или эквивалент из `stm02_is_*`) — обычное поле Net Profit из income statement.

**(г) Debt-to-Assets — строка `debttoassets` есть в файле, но не парсится.**
В столбце `Code` встречается значение `debttoassets`, но связанная строка в столбце `代码` имеет другой формат (без префикса метрики и суффикса `[rptDate]...`). Мой парсер её пропустил. При следующей выгрузке стоит либо забрать debt-to-assets отдельным полем, либо считать самостоятельно из Total Liabilities / Total Assets (оба поля заполнены хорошо).

**(д) 2025 год — 4.8 % заполнения, подтверждено.**
Только Total Revenue за 2025 имеет сколько-нибудь заметное покрытие (32 %). Все остальные метрики — 0–3 % за 2025. Решение: **удалить 2025** полностью, как и требует паспорт проекта (§2, временное окно 2012/2014–2024).

### 2.5 Что это означает для пайплайна

1. **Канонизация.** Прежде чем начать EDA и feature engineering, один раз пропустить файл через `parse_wind_file` и сохранить длинную панель в `data/processed/cn_active_panel.parquet` с колонками `ticker, name, year, metric, value`.
2. **Pivot к формату (ticker, year) → фичи.** После канонизации — стандартный pivot по metric, получая 319 × 12 × 15 ≈ 57 420 наблюдений в длинной форме или ~3 828 строк (ticker-year) × 15 фич в широкой.
3. **Фильтрация выборки.** Строки компаний-лет, где одновременно пусты Revenue, Assets и OCF, отбросить (drop-NA для активных компаний — паспорт §2.4).
4. **Временное окно.** Рекомендую брать 2018–2024 как основное окно (полная R&D, >80 % coverage по всем ключевым метрикам). 2014–2017 — опционально, для lag-фич и robustness-проверок.
5. **Метрики-блокеры.** Перезаказать из WIND: Net Profit, Total Debt / debt-to-assets, Interest Expense (если доступно). Без первых двух гипотеза H2 теряет одного из главных игроков (Net Income / Leverage).

## 3. Китай — делистнутые компании (разметка Y = 1)

**Файл:** `data/raw/china/bankrupt/Delisted stocks china.xlsx` (235 KB, 39 листов)

### 3.1 Что это
По переписке — 37 делистнутых китайских tech-компаний, для которых команда вручную собирала финансовые показатели и (предположительно) маркировала причину делистинга (банкротство / forced delisting / OTC / privatization / M&A / relisting). Это **источник правды для целевой переменной Y = 1** в паспортной постановке §2.2: «мы фильтруем и оставляем только реальные экономические дефолты».

### 3.2 Как это устроено

Структура оказалась существенно сложнее, чем предполагала переписка.

**Сводные листы (Sheet4, Sheet5):**
- `Sheet4` — 272 строки × 14 столбцов. Колонки: `代码`, `Name`, затем 12 столбцов вида «Cash Received / Operating Revenue [rptDate]2014年报» … 2025. **В этом листе 99.6 % ячеек пусты**, а среди тикеров — **индийские бумаги** (`3IINFOLTD.NS`, `500179.BO HCL INFOSYSTEMS`, `SILVERLINE TECHNOLOGIES` и др., всего 272 строки). Это остаточный «мусор» от более ранних экспериментов с индийской выборкой — к задаче делистинга Китая не относится.
- `Sheet5` — 4 строки, тоже пустая болванка под Total Revenue. Не используется.

**Индивидуальные листы (37 шт.):**
Названы по компаниям — `ACTIONS SEMICONDUCTOR-Financia1`, `AUTONAVI-Financial Summary1`, `LCT HOLDINGS-Financial Summary1`, `CHINACACHE-Financial Summary1`, … Структура одного листа:

```
Строка 0:  Short Name | <название>
Строка 1:  Start Year | 2014
Строка 2:  End Year   | 2026
Строки 3+: финансовые показатели по годам (73 строки, 2–6 столбцов)
```

То есть данные разложены **по компании-на-лист**, а не агрегированы в одной панели. Это нормальный формат выгрузки WIND по «экзотическим» тикерам (часто делистнутые/ADR/HK торгуются через отдельные экраны), но **нужен код, который обойдёт все 37 листов, распарсит каждый и склеит в единую панель**.

### 3.3 Критическая проблема: нет унифицированного `reason_of_delisting`

Аудит **не нашёл** в файле ни одного структурированного столбца, содержащего причину делистинга (ни `reason`, ни `outcome`, ни `status`, ни `cause`). Переписка команды предполагала, что «для каждой компании я дополнительно определил реальный исход» (Вадим, 2026-04-06), но в этой версии файла эти разметки не записаны явно — они, по-видимому, существуют либо:

- в отдельном файле/заметках, не загруженном в `data/raw/china/`;
- в прозе/комментариях внутри листов, что не структурировано;
- предполагается быть реконструированным вручную на данном этапе.

**Это блокер для Y = 1.** Без явной разметки «bankruptcy / forced / M&A / privatization / relisting» мы не можем применить паспортное правило (§1.3, §2.2): «отфильтровать делистинги, оставив только реальные дефолты». Если размечать всё подряд как Y = 1, модель будет учиться на шуме — часть «делистнутых» это SMIC (поглощён государством, перешёл на материковую биржу — не дефолт), SINA (privatization buyout — не дефолт), QIHOO 360 (privatization, затем вернулись на Shanghai — не дефолт), и т.д.

**Что нужно сделать:**
1. Найти/восстановить таблицу сопоставления 37 тикеров → причина делистинга. Если её нет в явном виде — собрать вручную по публичным источникам (HKEX delisting notices, SEC 15-12B/25-NSE, press releases, Wikipedia/Crunchbase).
2. Свести в CSV формата `ticker, company_name, delist_year, reason_code, is_default` с контролируемым словарём `reason_code ∈ {bankruptcy, forced_delisting, liquidation, privatization, m_and_a, voluntary, relisting, reg_sanction}` и правилом `is_default = 1` если `reason_code ∈ {bankruptcy, forced_delisting, liquidation}`.
3. Только после этого объединять с финансовой панелью для Y-разметки.

### 3.4 Список 37 компаний (из имён листов)

Для удобства фиксирую полный перечень — он пригодится на этапе ручной разметки:

```
ACTIONS SEMICONDUCTOR, AUTONAVI, LCT HOLDINGS, CHINACACHE,

```

(в файле 37 уникальных компаний + дубль `SINOTEL` = 38 листов «-Financial Summary» + Sheet4/Sheet5 = 39 листов всего, что совпадает с аудитом).

Уже на глаз из этого списка видно разнообразие исходов:
- **Явные дефолты:** LDK SOLAR (banktruptcy 2014), HANWHA Q CELLS (liquidation), CHINA SUNERGY, LINK MOTION (forced).
- **M&A / privatization (НЕ дефолт):** QIHOO 360 (privatization 2016 + возврат на Shanghai 2018), AUTONAVI (поглощение Alibaba), SINA (privatization 2021), MONTAGE TECHNOLOGY (acquired Intel → затем re-spin), SMIC (переведён на STAR Market Shanghai — не дефолт).

Это **значимо сдвигает дисбаланс классов**: 319 активных vs. ~18 дефолтов — соотношение ~18:1 (≈5 % положительного класса). Паспортная стратегия class-weights + отказ от SMOTE остаётся правильной, но нужно заранее понимать, что численно дефолтов меньше, чем ожидалось.

### 3.5 Парный файл — список 62 тикеров без финансов

**Файл:** `Delisted stocks_information_technology (2).xlsx` (13 KB, 1 лист Sheet1).
Структура: 13 строк × 63 столбца (1 ID-столбец + 62 тикера как столбцы). Содержит только мета-строки (Name, WIND Industry Name per year) — **финансовых показателей нет**. По переписке это «список из 62 компаний… как появится окно на доступ к базе данных — загружу такие же показатели».

То есть это **долг команды перед выборкой**: потенциально +25 делистнутых компаний к существующим 37, но пока без данных. Использовать в ML сейчас нельзя; держать как бэклог.

Кратко по content: все 62 тикера маркированы `WIND Industry Name(2024) = Information Technology` (кроме одного — BJL.SG/LCT HOLDINGS помечен Consumer Staples; плюс SOLA.L/SUNTECH как Utilities — это ошибки классификации WIND или смена бизнеса до делистинга).

## 5. Бразилия

### 5.1 Панельный файл

**Файл:** `data/raw/brazil/active/brazil_it_panel_improved.csv` (36 KB, 189 строк × 16 столбцов).

**Структура — образцовая:** длинный формат, одна строка = один company-year. Колонки:
`CNPJ_CIA, DT_REFER, year, Receita_Liquida, EBIT, Resultado_Bruto, Lucro_Liquido, Ativo_Total, Intangivel, Patrimonio_Liquido, Fluxo_Caixa_Operacional, DENOM_CIA, ROA, ROE, Debt_Equity, Current_Ratio`.

**Качество заполнения:**
- Все основные метрики — 0 % пропусков. Revenue, EBIT, Net Income, Assets, Intangibles, Equity, OCF, ROA, ROE — 100 % заполнены.
- Debt_Equity — 0.5 % пропусков (1 строка).
- **Current_Ratio — 100 % пропусков** (все NaN; видимо, расчёт сломался — нужны current assets + current liabilities, а их в явном виде в этой выгрузке нет).

**Размер:** 26 уникальных компаний (CNPJ) × ~8 лет = 189 наблюдений. Паспорт упоминает 29 компаний — расхождение небольшое (возможно, 3 компании были отфильтрованы на этапе чистки из-за полностью пустых строк).

### 5.2 Файл-контекст

**Файл:** `info_brazilian_companies.xlsx` (80 KB, 526 строк × 25 столбцов).

«Вселенная» компаний B3 (бразильская биржа) с NAICS-секторами, субсекторами и сегментами. Из этих 526 команда отобрала ~29 IT-компаний для основной панели. Для ML не нужен, но полезен как справочник для потенциального расширения выборки.

### 5.3 Вывод
Бразильские данные **чище всех остальных** по полноте, но:
- 26 компаний × 8 лет = слишком мало для устойчивого обучения ML (особенно ensemble).
- Нет размечанных дефолтов в самой панели — пришлось бы искать делистинги отдельно.
- Current_Ratio сломан → один из ключевых ликвидностных признаков недоступен.

Паспорт §2 и Introduction «Scope and Limitations» корректно исключают Бразилию. Можно оставить для descriptive cross-country сравнения в Chapter 3.5, но не для обучения.

## 6. ЮАР

Три файла: wide-панель, long-панель и мини-файл делистингов.

### 6.1 `south_africa_it_panel_wide.csv` (122 KB)
- 63 строки × **291 столбец** — очень широкая панель.
- Первые 3 столбца: `Ticker, Company, Year`. Остальные 288 — финансовые метрики в алфавитном порядке (`Accounts Payable, Accounts Receivable, Accumulated Depreciation, Additional Paid In Capital, …`).
- 63 строки для 15 уникальных компаний ≈ по ~4 года на компанию. Неравномерно.

### 6.2 `south_africa_it_panel_long.csv` (810 KB)
- 10 870 строк × 6 столбцов: `Ticker, Company, Year, Date, Metric, Value`.
- Длинный формат той же панели. **15 уникальных тикеров**, разные годы (есть 2021-12-31 и т.д.).
- Это тот же контент что и wide, только в тидимом виде.

Общая проблема: 288 метрик на 15 компаний означает **чудовищную разреженность**. Многие метрики существуют только для одной-двух компаний (например, `Allowance For Doubtful Accounts Receivable` встречается, но в большинстве строк NaN). Для ML такое неприменимо без радикального сокращения фич до ~10 базовых.

### 6.3 `south_africa_it_delistings.csv` (0.5 KB)
- **4 строки** × 9 столбцов. Да, всего четыре.
- Структура — образцовая: `Ticker, Company, Delist_Year, Event_Year, Type, Reason, Status, Source, is_default`.
- Из 4 строк: 3 помечены `is_default = 1` (EOH Holdings 2023, Mustek 2021, ISA Holdings 2022), 1 помечена `is_default = 0` (Altron — пережила restructuring).
- Хорошо, что есть **готовая логика target-переменной** (`is_default`) — единственный файл во всей выборке, где Y уже размечен.

### 6.4 Вывод по ЮАР
Объёмы **тривиальны**: 15 активных компаний, 3 дефолта. Для ML непригодно (даже Logit на 15 наблюдениях — anti-pattern). Паспорт корректно исключает ЮАР.

Но файл `south_africa_it_delistings.csv` можно использовать как **шаблон** разметки: его структура (`Ticker, Company, Delist_Year, Event_Year, Type, Reason, Status, Source, is_default`) — ровно то, чего не хватает китайскому файлу делистингов (см. §3.3). Рекомендую воспроизвести эту же схему для китайских 37 компаний.

## 7. Сквозные технические замечания

### 7.1 Три разных формата WIND-выгрузок
- Китайская активная панель: тикеры в **столбцах**, метрики-годы в **строках**.
- Индийская панель: тикеры в **строках**, метрики-годы в **столбцах**.
- Китайский делистинг: **по-компании-на-лист**, с общей схемой внутри листа.

Это означает, что единый загрузчик нужен адаптивный (switch по heuristic — кол-во столбцов vs кол-во строк, наличие префикса `[rptDate]` в заголовках).

### 7.2 Кодировка и локали
Китайские файлы используют `代码` (UTF-8) как первый столбец и `YYYY年报` (UTF-8) в годах. Важно: никаких `encoding='cp1251'` — только UTF-8 при чтении. Pandas по умолчанию справляется, но при записи в CSV указывать `encoding='utf-8-sig'`.

### 7.3 Единицы измерения
WIND выдаёт суммы в **миллионах юаней** (`[unit]1M`). При склейке с другими странами (Россия — рубли, Бразилия — реалы, ЮАР — рэнды) необходима конвертация в единый numéraire (USD на конец года, либо деление на CPI → реальные значения). Но для TTC-стадии это некритично — модель всё равно работает с ratio-фичами (ROA, Intangibles/Assets, OCF/Liab и т.д.), которые инвариантны к масштабу.

### 7.4 Классовый дисбаланс — уточнение цифр
Паспорт ссылается на «~319 vs ~37» как на соотношение 8.6:1. С учётом того, что из 37 делистингов реальных дефолтов будет ~15–22 (см. §3.4), **фактическое соотношение в обучающей выборке для Китая, скорее всего, составит 14:1 — 21:1** (6–7 % positive class). Для cost-sensitive подхода это означает `class_weight = {0: 1, 1: ~16}` как стартовая оценка (обратная частоте). Паспортный отказ от SMOTE (§2.4) остаётся в силе.

### 7.5 Вопрос временной согласованности (leakage)
Делистнутая компания по определению перестаёт сдавать отчётность в некоторый год T_del. Для неё последние фактические отчётности — T_del - 1 или T_del - 2. При построении «момента предсказания за год до дефолта» нужно аккуратно выбирать T_pred = T_del - 1 и не допускать, чтобы активные компании сравнивались с дефолтными на одних и тех же годах (иначе модель подхватит тренд «свежие годы → активные, старые → дефолтные»). Это стандартный survival/event-study issue, который нужно учесть в train/test split.
